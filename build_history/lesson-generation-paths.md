# Dreizunge — how lessons are generated (v42)

A detailed reference for every path that produces a lesson, from the trigger down
to persistence. Companion diagram: `lesson-generation-paths.svg` / `.png` (source `.dot`).

---

## 0. One-line mental model

> **trigger → endpoint (async job) → [story: provided or generated] → meta (title+emoji, translated to source lang) → dispatch by `lessonFormat` → one or more generators → `PROMPTS.<section>` → LLM → validate/retry → persist + chain into storyline**

…with one exception: **add-lesson** skips the story-generation and meta stages and calls a single generator directly on an existing topic.

---

## 1. Entry points and the async job model

Five user-facing triggers funnel into **three** generation endpoints in `server.js`:

| Trigger | Endpoint | Story source |
|---|---|---|
| Web generation form | `POST /api/generate` | model-generated |
| `generate-lessons.js` (batch) | `POST /api/generate` | model-generated |
| `translate-lessons.js` (batch) | `POST /api/generate` | **provided** (`translategemma`-translated template story) |
| Web “PDF book” (from pasted text) | `POST /api/generate-book` | **provided** (one chunk per chapter) |
| Web ➕ add-lesson | `POST /api/lessons/add-lesson` | the topic’s **existing** story |

(`POST /api/qc` is *not* generation — see §11.)

**Async model.** Every endpoint creates a job (`newJob()`), returns `{ jobId }` immediately (HTTP 202), and runs the work in the background. Progress is reported with `jobStep(jobId, …)`; completion with `jobDone(jobId, data)`; failure with `jobFail(jobId, err)`. Clients/scripts poll `GET /api/job/:id` until `status` is `done` or `error`. A per-topic lock (`generatingTopics`) prevents two concurrent generations on the same topic (HTTP 429).

> ⚠ The `/api/generate` job result does **not** carry the new topic’s stable `id` (it is assigned to the stored copy after generation). Tooling resolves the id afterward from `/api/lessons`. See `translate-lessons.js`.

---

## 2. What gets produced (data shapes)

A **topic** (`tp_…`) is one story unit:

```
{ id, topic, topicEmoji, lang, srcLang, story, storyLang, difficulty,
  storyLen, storyStyle, continuedFromId, lessons: [ … ] }
```

A **lesson** is one entry in `lessons[]`. Its shape depends on type:

- **standard** — `{ vocab: [{target, source}], sentences: [{target, source}] }`
- **grammar** — `{ type:'grammar', grammar: [{target, source, gender, article, plural}] }`
- **conjugation** — `{ type:'conjugation', … per-verb paradigms }`
- **synonyms** — `{ type:'synonyms', words: [{base, gloss, synonyms[], antonyms[], homophones[]}] }`
- **error_hunt** — corrupted sentences + the corrections
- **math** — ordering + calculation exercises

Individual vocab/sentence items may also carry a QC flag: `qc: { sug, field, at }` (§11).

---

## 3. The core pipeline: `generate()`

`/api/generate` and `/api/generate-book` both call
`generate(topic, lang, srcLang, difficulty, continuedFrom, storyLen, jobId, userOpts)`.

### 3.1 Inputs (`userOpts`)

`userStory`, `userTranslation`, `userDialect`, `storyStyle`, `lessonFormat`,
`reinforcePrior`, `vocabMode`, `useFullChain`, `userStoryLang`, `prevStoryTopic`,
`mathInstruction`.

### 3.2 Stage 1 — Story

- **If `userStory` is provided** (own text, a translation, or a book chunk): it is used verbatim as the story. `storyLang` is `userStoryLang` (default `target`).
- **Otherwise**: the story is generated by the model. System prompt = `sysStory(lang, hasPrev, storyLen, dialect, style)` (built from `PROMPTS.story` + `PROMPTS.storyStyles`). If continuing a chain, the **previous chapter’s story** is prepended as context — either the full text (`useFullChain`) or the last `OLLAMA_MAX_PREV_STORY` chars. Word count is steered by `storyLen` (server floor ≈ 100 words for generated stories; PDF chunking honors lower values).

### 3.3 Stage 2 — Meta (title + emoji, then translate)

`PROMPTS.meta` produces `{ topic, topicEmoji }` from the story. Then `metaTranslation`
**translates the title into the source language** so it reads in the learner’s language.
This is why the **saved topic name often differs from the string that was sent** — name-based
matching in external tooling is unreliable; match by id/lang instead.

If meta generation/parse fails, it falls back to the input topic + 📚.

### 3.4 Stage 3 — Dispatch by `lessonFormat`

| `lessonFormat` | Generator(s) | Prompt section |
|---|---|---|
| `standard` | `generateOneLesson` | `PROMPTS.vocab` |
| `grammar` | `generateGrammar` | `PROMPTS.grammar` |
| `conjugation` | `generateConjugation` | `PROMPTS.conjugation` |
| `synonyms` | `generateSynonyms` | `PROMPTS.synonyms` |
| `error_hunt` | `generateErrorHunt` | `PROMPTS.errorHunt` |
| `math` | `generateMath` (no LLM) / `generateMathLLM` | `PROMPTS.math` (LLM only) |
| `all_types` | standard **+** grammar **+** conjugation **+** error_hunt | the above |

Single-format generation produces **one** lesson; `all_types` produces **four** (and tolerates failure of the extra three — they’re skipped with a warning, the standard lesson still lands).

## 4. The lesson generators in detail

Common pattern: each generator builds a system prompt (`sysXxx`) and a user message
from `PROMPTS.<section>` via `fillPrompt`, calls `callLLMLesson` (the lesson model),
parses JSON (with a brace-extraction fallback), **validates**, and **retries up to 3×**
(`MAX_ATTEMPTS = 3`) before throwing. Each returns `{ lesson, tokens }`.

### 4.1 `standard` → `generateOneLesson`

Produces a vocab + sentences lesson. The user message = `PROMPTS.vocab.user` plus:

- **storyHint** — an **excerpt of the story** (first ~1200 chars), wrapped by a *story-language-aware* variant: `storyHint_target` / `storyHint_source` / `storyHint_other` depending on whether the story is in the target language, the source language, or another. So the vocab is drawn from the actual story text.
- **prevHint** — within-topic prior vocab (avoid repeating earlier lessons of the same topic).
- **chainHint** — cross-chapter vocab from `chainVocab` (see §5).
- `PROMPTS.vocab.suffix`.

**Word-form rule (v42):** vocab items must be **base/citation forms** (infinitive verbs, singular nouns with article where applicable, base adjectives). Sentences are taken from / grounded in the story.

> `PROMPTS.vocabFromText` and `PROMPTS.vocabTable` are alternate prompt sets (text-extraction and user-supplied-table parsing) used by other flows; the standard generate path uses `PROMPTS.vocab` with the embedded excerpt.

### 4.2 `grammar` → `generateGrammar(topic, lang, srcLang, diff, jobId, opts)`

Generates ~10 nouns with **gender, article, and plural**. User message =
`PROMPTS.grammar.user` plus:

- **storyHint** — `extractKeywords(opts.story, 6, lang)` → “prefer these where topic-relevant”. **Only present if `opts.story` is supplied.**
- **priorNouns** — from `chainVocab.nouns` (≤15): *reinforce* (“prefer”) or *extend* (“AVOID these”).

Validated by `validateGrammarItems` (rejects malformed/wrong-language items). **Word-form rule (v42):** grammar **focuses on the derived forms used in the story** (plurals, marked forms) — the counterpart to the vocab base-form rule.

### 4.3 `conjugation` → `generateConjugation(...)`

Generates ~5 verbs with their conjugation paradigm. Same structure as grammar:
`PROMPTS.conjugation` + storyHint (`extractKeywords` of `opts.story`, if present) +
prior **verbs** from `chainVocab.verbs` (reinforce/extend). **Focuses on the conjugated
forms used in the story** (v42).

### 4.4 `synonyms` → `generateSynonyms(...)`

Grouped study lesson: for each base word, `{ synonyms[], antonyms[], homophones[] }`
(synonyms required; antonyms/homophones optional). Prefers words from the story
(`storyKeywords`) and the reinforce list (`chainVocab.words` via `priorWords`). Server
validates each group (≥1 synonym, strips self-references, dedupes, caps 4/3/2). Homophones
are optional because in the quiz they’re used only as trap distractors.

### 4.5 `error_hunt` → `generateErrorHunt(story, lang, diff, jobId, priorVocab)`

Takes the **story** and produces sentences with deliberate errors plus corrections
(`sysErrorHunt` / `PROMPTS.errorHunt`). Optional `priorVocab` hint biases errors toward
chain words. Requires a story (it operates on the story text).

### 4.6 `math` → `generateMath` / `generateMathLLM`

- **`generateMath(story, diff, ops)`** — **no LLM**: `extractNumbers(story)` then builds ordering + calculation exercises deterministically from the numbers found in the story.
- **`generateMathLLM(lang, srcLang, diff, instruction, jobId)`** — used when a free-text math instruction is given; goes through `PROMPTS.math`.

### 4.7 `all_types`

Runs the standard vocab lesson, then appends grammar, conjugation, and error_hunt as
additional lessons of the same topic. Failures in the extra three are logged and skipped.

## 5. Cross-chapter continuity: `vocabMode` and chain vocab

`vocabMode` (`neutral` | `reinforce` | `extend`) controls whether and how vocabulary from
**earlier chapters of the same storyline** influences the new lesson. It is independent of
the story; it’s about cross-chapter continuity.

- **`neutral`** (default) — chain vocab is **not collected**; the lesson has no cross-chapter awareness. Words come from the topic/story + the model.
- **`reinforce`** — `collectChainVocab(parent)` gathers prior words; they’re passed as a **soft “prefer these”** hint (reuse earlier vocabulary so it sticks).
- **`extend`** — the same prior words are passed as an **exclusion list (“AVOID these”)** to force fresh vocabulary.

Preconditions: chain vocab is non-empty only when the topic **has a parent**
(`continuedFromId`) **and** the mode isn’t `neutral`. `collectChainVocab` returns
`{ words, nouns, verbs }`, consumed by standard (`words`), grammar (`nouns`),
conjugation (`verbs`), and synonyms (`words`). In `generate()` the legacy
`reinforcePrior:true` maps to `reinforce`.

---

## 6. Word-form conventions (base vs derived)

A deliberate division of labour across lesson types (v42 prompts):

- **Vocab** lessons teach **base/citation forms** (infinitive, singular, base adjective). `vocabFromText` lemmatizes story words even when the story shows an inflected form; sentences stay verbatim.
- **Grammar** and **conjugation** lessons instead **focus on the derived forms that appear in the story** (plurals/marked forms; conjugated forms).

This steer depends on the generator receiving story context (a story excerpt for vocab, or `extractKeywords(story)` for grammar/conjugation). Without a story, the “derived forms from the story” instruction has nothing to anchor to and the model falls back to topic-driven choices.

## 7. The add-lesson path (`POST /api/lessons/add-lesson`)

Adds **one** lesson to an existing topic. It is structurally different from `generate()`:

- It **requires the topic to already have a story** (400 otherwise) and **reuses that stored story** — it does **not** generate a new story and does **not** run the meta/title step.
- It resolves `vocabMode` the same way (`vocabMode` or `reinforcePrior`, default `neutral`) and builds `chainVocab` from the parent when applicable.
- It dispatches to a **single** generator by `fmt`:

| `fmt` | Receives the topic’s story? | Notes |
|---|---|---|
| `standard` | ✅ (via `lessonOpts.story`) | story excerpt → vocab |
| `synonyms` | ✅ | story keywords |
| `error_hunt` | ✅ | operates on the story |
| `math` | ✅ | numbers from the story (or instruction → `generateMathLLM`) |
| `grammar` | ✅ **(fixed in v42)** | now passed `story` → `extractKeywords` |
| `conjugation` | ✅ **(fixed in v42)** | now passed `story` → `extractKeywords` |

**v42 fix:** grammar and conjugation added via the ➕ button now receive `saved.story`, so
their prompts include the story-keyword hint (verified: the grammar prompt gains a
“Context nouns/themes from the story (prefer these…)” line). Previously these two were the
only generators in add-lesson that ignored the story, which made the “story-derived forms”
steer inert for them.

The new lesson is appended to the topic’s `lessons[]` and the store is saved.

---

## 8. The book path (`POST /api/generate-book`)

For generating a whole storyline from pasted text:

1. Split the text into chapter-sized **chunks** (chunk size honors the story-length control).
2. Resolve the optional parent (id-or-name).
3. **Loop `generate()` once per chunk**, passing `userStory = chunk` and chaining each chapter to the previous (`continuedFrom`), so they form one storyline.

Runs as its own book job (`bookJobs`), survives a client refresh, and is dropped only on server restart.

---

## 9. Batch scripts (external entry points)

- **`generate-lessons.js`** — drives `POST /api/generate` across many language pairs with fresh, model-generated stories.
- **`translate-lessons.js`** — takes an exported storyline as a template, translates each chapter’s story with `translategemma` (Ollama) → sends it as `userStory` so the server rebuilds correct lessons for the new target language; chains chapters into a storyline; resolves topic ids from `/api/lessons` (since the job response omits them).
- **`qc-lessons.js`** — standalone QC (does not generate). See §11.

---

## 10. Persistence and storyline syncing

- `generate()` → returns the topic entry → the job handler `upsert`s it and calls `_syncStorylineForTopic`, which rebuilds the storyline from the `continuedFromId` chain (chapters group automatically and incrementally).
- `add-lesson` → appends the lesson to the existing topic and `saveStore`.
- New topic ids are random-numeric (`tp_…`), satisfying the client’s `/^tp_\d+$/` check.

---

## 11. QC (separate flow — not generation)

`POST /api/qc` proofreads existing lessons; it never generates. Scope: `{storylineId}` |
`{topicId}` | `{topicId, lessonIdx}`. For each vocab/sentence pair it asks the translation
model to verify **both** the target text (spelling/grammar/capitalization) **and** the
translation, writing `qc:{sug, field, at}` onto flagged items (field = `target` or
`source`). The editor’s ✓ applies the fix to that field; ✕ clears the flag. The standalone
`qc-lessons.js` performs the same checks from the command line.
